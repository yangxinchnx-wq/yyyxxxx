## 1. 架构核心概念

### 1.1 角色分工

| 角色 | 实际模型 | 做什么 |
|---|---|---|
| **主模型** | SettingsModal 配的"主模型"槽位 | **调度者**：拆任务 → 分给副模型 → 汇总结果 → 出最终回答 |
| **副模型** | SettingsModal 配的、副模型开关勾选的、`models[i].enabled===true` 的 | **工作者**：接收子任务、干活、出结果 |
| **AI 社会** | 后端编排 | 监督副模型输出，提升最终质量 |

### 1.2 AI 社会四角色

| 角色 | 英文 | 职责 |
|---|---|---|
| **打分** | Scorer | 给每个副模型输出打分（质量分、置信度分） |
| **评判** | Judge | 对比各副模型输出，挑最优，淘汰劣质 |
| **审计** | Auditor | 找漏洞、错误、偏差 |
| **交付** | Deliverer | 整合所有轮次，输出最终回答 |

### 1.3 四种运行模式 × 资源矩阵

| 模式 | 主模型 | AI 社会 | 副模型 |
|---|---|---|---|
| **普通 (normal / 安全)** | ✅ 直接出回答 | ❌ 不参与 | ❌ |
| **性能 (performance / 半自动)** | ✅ 调度 | ✅ 决策+审计+交付（无打分/评判） | ✅ 仅用户勾选且 SettingsModal 启用的 |
| **专家 (expert / 全自动)** | ✅ 调度 | ✅ **最小化**：单角色"审计"做三步（找漏洞 → 修补 → 补充） | ✅ 仅勾选且启用的 |
| **极致 (ultimate / 全自动)** | ✅ 调度 | ✅ 完整四角色，不惜资源 | ✅ 全部已启用副模型 |

UI 顶部四个开关位置：`ChatPanel.tsx:1273-1276`

### 1.4 智能动态启用（**仅高级模式**）

**触发范围**：仅 **性能 / 专家 / 极致** 模式（普通模式不参与反问）

逻辑：AI 社会评估时如果发现"启用某个 `enabledInSettings=false` 的副模型能提升 50% 以上收益"，**不直接启用**，而是 **SSE 推送反问事件**，前端弹 confirm 让用户决定。

**普通模式跳过反问的理由**：普通模式没有 AI 社会参与，没有"未启用副模型可贡献收益"这种评估，自然没有反问事件。

## 2. 持久化层新增

### 2.1 新 localStorage 键：`cherry_model_provider_map_v1`

**用途**：建立"模型显示名 → Provider 凭据"的映射，让 ChatPanel 能挑出 baseUrl / apiKey / model id。

**数据结构**：
```ts
type ModelProviderEntry = {
  providerId: string;         // "openai" / "deepseek" / "custom_xxx"
  providerName: string;       // "OpenAI" / "DeepSeek"
  baseUrl: string;            // 上游 API 地址
  apiKey: string;             // 加密字符串（用现有 encryptSecret）
  model: string;              // 实际调用的 model id（如 "gpt-4o"）
  enabledInSettings: boolean; // 来自 SettingsModal.models[i].enabled
};

type ModelProviderMap = Record<string /* displayName */, ModelProviderEntry>;
```

**示例**：
```json
{
  "GPT-4o": {
    "providerId": "openai",
    "providerName": "OpenAI",
    "baseUrl": "https://api.openai.com/v1",
    "apiKey": "enc:v1:xxx:yyy",
    "model": "gpt-4o",
    "enabledInSettings": true
  },
  "DeepSeek-V3": {
    "providerId": "deepseek",
    "providerName": "DeepSeek",
    "baseUrl": "https://api.deepseek.com/v1",
    "apiKey": "enc:v1:aaa:bbb",
    "model": "deepseek-chat",
    "enabledInSettings": true
  }
}
```

### 2.2 加密时机：**用户输入 key + 测试通过后才加密**

apiKey 的状态机：

```
[未输入]  → 用户输入明文  → [明文，未测试]  → 测试通过  → [enc:v1:...] 加密写盘
                                  ↘ 测试失败   → 仍是 [明文，未测试]，localStorage 同步明文写盘
```

**关键变化（与 §2.1 早期版不同）**：
- ❌ 不再"providers 变化时立刻加密"
- ✅ 改成"providers 变化时始终**明文**写盘"
- ✅ 在 `testProviderConnection` 的 `data?.success` 分支里，**单独调用一次"加密该 provider"**

**为什么这样设计**：
- 测试失败的 key 不应被加密（用户可能继续调试/重输），加密反而丢失明文带来风险
- 测试通过 = key 已经被验证为"能用"，此时加密才有意义
- 用户不主动测试就明文存盘，是用户自己的选择（localStorage 本就是用户机器上的）

**SettingsModal.tsx 改动点**：

1. **删除现有 line 327-340 的 useEffect**（每次 providers 变化就加密写盘的）
2. **新增 useEffect**（明文写盘）：
   ```ts
   useEffect(() => {
     localStorage.setItem('cherry_providers_v2', JSON.stringify(providers));
     window.dispatchEvent(new CustomEvent('providers_updated'));
   }, [providers]);
   ```
3. **修改 `testProviderConnection` 的成功分支**（line 554-560 区域）：
   ```ts
   if (data?.success) {
     // 测试通过 → 加密该 provider 的 apiKey 并写盘

     setProviders(prev => prev.map(p => p.id === providerId ? {
       ...p, status: 'success', delay: data.latency, errorMessage: undefined, apiKey: encKey
     } : p));
   } else {
     // 测试失败 → 保持明文，方便用户重输
     setProviders(prev => prev.map(p => p.id === providerId ? {
       ...p, status: 'failed', errorMessage: data?.error || '连接失败'
     } : p));
   }
   ```
4. **删除现有 line 346-362 的 mount 解密 useEffect**（不再需要，启动时直接读明文即可；如果是迁移旧数据则按需保留一次解密逻辑）
5. **现有 line 262-267** 的"内置演示 key 替换"逻辑保持不变（仅过滤特定内置示例值）
6. **新增 `cherry_model_provider_map_v1` 写入逻辑**：放在 `testProviderConnection` 成功分支后，只有该 provider 的 apiKey 已加密才写映射

**字段加密规则不变**：
- 仅 `apiKey` 字段加密（用现有 `encryptSecret`）
- 其余字段明文
- `enc:v1:` 前缀判断逻辑保持不变

### 2.3 读出时机（App.tsx mount + 事件）

- 启动时 `App.tsx` 读 `cherry_providers_v2` → 直接拿到 apiKey（明文，未测试）或（加密，已测试）→ 写入 state

### 3.4 `society/orchestrator.ts`

导出三个编排函数：

```ts
export async function runUltimate(req, sse): Promise<void>
export async function runExpert(req, sse): Promise<void>
export async function runPerformance(req, sse): Promise<void>
```

**统一入参**（与 §3.5 端点入参对齐）：
```ts
type OrchestratorReq = {
  mode: 'performance' | 'expert' | 'ultimate',
  mainProvider: { baseUrl, apiKey, model },
  subProviders: [{ baseUrl, apiKey, model }],         // 已勾选且启用的副模型
  candidateProviders: [{ displayName, providerName, modelName }],  // 候选（脱敏，不含 apiKey）
  query: string,
  history: [{ sender, content }],
  fileContext?: { name, content }
}
```

**统一入参 `candidateProviders`（脱敏）**：前端把所有"已配置但未启用"的副模型脱敏后传过来：
- `displayName` — UI 显示名（GPT-4o / DeepSeek-V3）
- `providerName` — Provider 名（OpenAI / DeepSeek）
- `modelName` — model id
- **不含** baseUrl / apiKey

**为什么脱敏**：用户的未启用 Provider apiKey 不应离开用户机器。后端在阶段 0 评估时只能基于"模型名 + Provider 类型 + query 上下文"做启发式评估，精度有限但符合安全边界。

#### 阶段 0：反问评估（性能/专家/极致都执行，普通跳过）

**前置判断**：先看 `req.enableDecision` 有没有：
- 有 → 用户已决策 → 跳过阶段 0，直接进入主流水线
- 无 + 模式是普通 → 跳过阶段 0，直接进入主流水线
- 无 + 模式是性能/专家/极致 → 执行阶段 0

**阶段 0 逻辑**（无 `enableDecision` + 高级模式）：

```
输入: candidateProviders（脱敏）+ query
启发式评估: 用主模型做一次轻量调用（max_tokens=200）
            问主模型: "在 query 场景下，{modelName} ({providerName}) 
                     是否能比当前 subProviders 提供 ≥50% 收益？"
            主模型回答 yes → 收益 ≥ 50% → 推 suggest_enable
            主模型回答 no / 不可信 → 跳过
输出: 发完所有 suggest_enable 后立即 end 流
```

**为什么用主模型做启发式**：后端没拿到 candidate apiKey，无法实际调用候选模型。让主模型基于"模型名 + Provider 名 + query 上下文"做经验判断即可。精度有限但足够反问决策。

**用户响应机制**（前端负责）：
- 后端发完 `suggest_enable` 后**立即 end 流**（不等待）
- 前端 streamSSE 解析到 `suggest_enable` → 弹 confirm 收集用户决策
- 用户决策后**前端重新发起一次** `/api/ai/chat` 请求，body 多带 `enableDecision: { candidateName, accept: boolean }`
- 后端看到 `enableDecision` → 把该 candidate 加进 `subProviders` → 继续主流水线
- 用户 30s 不响应 → 前端兜底**重新发起一次**请求，`enableDecision: { accept: false }`（或省略该字段），后端用原 `subProviders` 继续

**协议总结**：两次完整 SSE 请求，不是双工。前端 merge 两个流的 `phase` 事件到同一个 UI state。

#### 极致模式流水线（5 阶段，串行 SSE 推送）

```
阶段1: 主模型 dispatch → 拆 N 个子任务 → 分配给 N 个副模型并行
       SSE event: phase=dispatch, data={subtasks:[...]}
阶段2: Scorer 打分（一次 LLM 调用）
       SSE event: phase=score, data={scores:[...]}
阶段3: Judge 选定（一次 LLM 调用）
       SSE event: phase=judge, data={chosen:[idx], reasoning}
阶段4: Auditor 审计（一次 LLM 调用）
       SSE event: phase=audit, data={findings:[...]}
阶段5: Deliverer 整合（一次 LLM 调用，stream:true）
       SSE event: phase=deliver, data={delta:"..."}（多次）
       SSE event: phase=done, data={reply, audit}
```

#### 专家模式流水线（3 步，最小化）

```
阶段1: 主模型 dispatch → 副模型并行（如果启用）
阶段2: Auditor 单 prompt 三段式（找漏洞 → 修补 → 补充），流式
       SSE event: phase=audit, data={delta:"..."}（多次）
阶段3: 主模型汇总
       SSE event: phase=done, data={reply, audit}
```

#### 性能模式流水线（3 阶段，省资源）

```
阶段1: 主模型 dispatch → 副模型并行
阶段2: 主模型直接汇总（无打分/评判）
阶段3: Auditor 单轮审稿
       SSE event: phase=audit, data={findings:[...]}
       SSE event: phase=done, data={reply, audit, suggestions}
```

#### `streamSSE` 工具函数

```ts
async function streamSSE(url: string, body: any) {
  const res = await fetch(url, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(body)
  });
  if (!res.ok) { /* 错误处理 */ return; }
  const reader = res.body.getReader();
  const decoder = new TextDecoder();
  let buffer = '';
  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    buffer += decoder.decode(value, { stream: true });
    const events = buffer.split('\n\n');
    buffer = events.pop() ?? '';
    for (const evt of events) {
      const eventMatch = /^event:\s*(\w+)/m.exec(evt);
      const dataMatch = /^data:\s*(.+)$/m.exec(evt);
      if (!eventMatch || !dataMatch) continue;
      const phase = eventMatch[1];
      const data = JSON.parse(dataMatch[1]);
      handlePhase(phase, data);
    }
  }
}
```

#### `handlePhase` 状态机

| phase | 动作 |
|---|---|
| `dispatch` | 更新 `subtasks` state |
| `score` | 更新 `scorePanel` state |
| `judge` | 更新 `chosen` state |
| `audit` | 更新 `auditPanel` state |
| `reply` / `deliver` (delta) | 追加到当前 assistant 消息 |
| `done` | 关闭 `isGenerating` |
| `warn` | 显示警告横幅 |
| `suggest_enable` | 弹 confirm，用户确认后带新 providers 重发 |
| `error` | 显示错误，停止流 |

#### 新增 UI 子组件

| 组件 | 何时显示 | 内容 |
|---|---|---|
| `WorkerOutputsPanel` | 性能 / 极致 模式 | 每个副模型独立卡片，显示其输出 |
| `ScorePanel` | 仅极致模式 | 表格：workerIdx / modelName / score / reason |
| `JudgePanel` | 仅极致模式 | chosen + reasoning |
| `AuditPanel` | 启用 AI 社会的模式 | findings 列表，按 severity 配色 |
| `FinalReply` | 始终 | 最终回答流式追加 |
| `SuggestEnableDialog` | 收到 `suggest_enable` 时 | confirm 弹窗："启用 X 模型可提升约 Y%，是否开启？" |

## 5. 错误处理矩阵

| 场景 | 后端响应 | 前端处理 |
|---|---|---|
| 主模型 apiKey 为空 | 400 `{error:"mainProvider.apiKey 必填"}` | toast |
| 副模型全部未启用 | 200 SSE `phase=warn, data={msg:"未启用副模型，降级为普通模式"}` | 显示警告横幅 |
| 上游超时 60s | SSE `event: phase=error, data={code:"TIMEOUT"}` | 中止流，显示错误 |
| 主模型 401/403 | SSE `event: phase=error, data={code:"AUTH_FAILED"}` | toast "请检查 API key" |
| SSE 连接中断 | — | 显示"连接中断，点击重试"按钮 |
| 上游响应非 JSON | 502 `{error:"上游响应不是 JSON"}` | toast |
